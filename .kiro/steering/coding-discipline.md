# 编码纪律规则

基于 Andrej Karpathy 的 LLM 辅助编程实践总结，以下规则强制执行。

## 1. 先读后写

- 动手前必须读取即将修改的文件，是读，不是扫
- 复用项目中已有的模式、导入和依赖（例如项目用 `fetch` 就不要引入 `axios`）
- 找不到已有模式时，问用户，不要猜

## 2. 先想后写

- 动手前明确：做什么、不做什么、假设是什么
- "加个验证" 有五种理解，先说清楚选了哪一种
- 遇到真正不确定的地方就停下来问，不要用"看起来合理"的代码填充不确定性

## 3. 最小实现

- 只写解决当前问题的最少代码
- 不做预防性抽象——只有出现真实重复时才抽象（至少 copy-paste 两次再考虑）
- 不处理不可能发生的错误
- 不配置化——除非有真实的配置需求
- 判断标准：如果唯一的理由是"以防将来需要"，就是过度设计

## 4. 外科手术式变更

- diff 尽可能小，只改任务要求的部分
- 不触碰任务范围之外的代码
- 匹配已有代码风格
- 不做格式化——格式化把真正重要的 3 行埋在无关的 300 行里
- 每一行变更都必须能用任务来justify；"顺便改了"的行，撤回

## 5. 验证

- 修 bug 时：先写失败测试 → 看它失败 → 再修复
- 测试行为，不测 setter 赋值之类的琐碎实现
- 如果某段代码难以测试，这是设计问题的信号，不是跳过测试的许可

## 6. 目标驱动

- 每个任务在写代码前先有验收标准
- 错误示范："加验证"
- 正确示范："拒绝缺失或格式错误的 email，返回 HTTP 400 + 明确提示，测试两种情况"
- 多步任务先陈述计划，让用户能在错误方向投入之前纠偏

## 7. 调试

- 出问题时：调查，不要猜
- 读完整错误信息和堆栈
- 先复现，再修改
- 每次只改一个变量
- 不要用 null check 掩盖意外的 `null`——搞清楚它为什么是 null

## 8. 依赖管理

- 每个依赖都是你不可控的永久代码
- 加之前先确认项目或标准库是否已经能做到（例如 `crypto.randomUUID()` 而非 uuid 包）
- 加依赖时说明原因，让选择可见而非静默混入 manifest

## 9. 沟通

- 说清楚做了什么、为什么这么做，不只是丢一块代码
- 即使完全按照要求做了，也要标出隐患
- 精确表达不确定性
- 好："不确定这个库是否支持 streaming"（指出需要验证的点）
- 坏："应该能用"（什么信息都没有）

## 10. 常见失败模式（自检清单）

遇到以下模式立即停止，不要硬推：

| 模式 | 表现 |
|------|------|
| Kitchen Sink | 改一个功能顺便重构半个项目 |
| Wrong Abstraction | 还没出现重复就抽象 |
| Optimistic Path | 只实现 happy path，忽略失败场景 |
| Runaway Refactor | 一个修复级联到十几个文件 |
